The Case Against Academic Identity Providers

A Data-Driven Analysis of Why SAML Federation Has Failed Higher Education

Zeus Auth Research Division ‚Ä¢ January 2026

Abstract

This analysis examines the structural, technical, and operational failures of SAML-based academic identity federation systems, including Shibboleth, OpenAthens, and eduPerson attribute schemas. Through examination of protocol complexity metrics, implementation failure rates, developer experience data, and total cost of ownership analysis, we demonstrate that these systems represent a significant impediment to educational technology adoption. We propose modern OAuth 2.0/OIDC alternatives as a path forward.

1. Introduction: The Federation Problem

Academic identity federation emerged in the early 2000s as a solution to a genuine problem: how to enable single sign-on across institutional boundaries. The Security Assertion Markup Language (SAML) 2.0, ratified in 2005, became the foundation for systems like Shibboleth (Internet2) and OpenAthens (Jisc).

Two decades later, these systems have become monuments to technical debt. What was designed for a pre-mobile, pre-cloud, pre-API world now struggles to meet the needs of modern educational technology.

Sources: InCommon Annual Report 2024, EDUCAUSE IT Issues Survey 2024

2.2 XML Signature Wrapping Attacks

The complexity of XML canonicalization and signature verification has led to a class of vulnerabilities known as "XML Signature Wrapping" (XSW) attacks. Research has documented over 20 variants of XSW attacks affecting SAML implementations.

CVE History: SAML Libraries

  • CVE-2017-11427 - OneLogin SAML - Signature bypass
  • CVE-2018-0489 - Shibboleth - XML entity expansion DoS
  • CVE-2018-7340 - SimpleSAMLphp - Authentication bypass
  • CVE-2019-3465 - xmlsec1 - Signature verification bypass
  • CVE-2020-5390 - PySAML2 - Signature wrapping attack
  • CVE-2021-21239 - Python SAML - Authentication bypass
  • CVE-2022-44457 - Mendix SAML - Authentication bypass
  • CVE-2023-45133 - Keycloak SAML - Signature bypass

The fundamental problem: XML's flexibility makes it difficult to validate correctly.

3. The eduPerson Attribute Disaster

The eduPerson schema, maintained by EDUCAUSE, defines attributes for representing people in higher education. What began as a reasonable standardization effort has become a case study in committee-driven complexity.

3.1 The OID Problem

eduPerson attributes are identified by Object Identifiers (OIDs), a hierarchical naming scheme from X.500 directory services. These OIDs are human-hostile:

OID Meaning OAuth Equivalent
urn:oid:1.3.6.1.4.1.5923.1.1.1.6 eduPersonPrincipalName email
urn:oid:1.3.6.1.4.1.5923.1.1.1.9 eduPersonScopedAffiliation groups or custom claim
urn:oid:1.3.6.1.4.1.5923.1.1.1.10 eduPersonTargetedID sub
urn:oid:1.3.6.1.4.1.5923.1.1.1.13 eduPersonUniqueId sub
urn:oid:2.5.4.42 givenName given_name
urn:oid:2.5.4.4 sn (surname) family_name
urn:oid:0.9.2342.19200300.100.1.3 mail email
"Quick, what's urn:oid:1.3.6.1.4.1.5923.1.1.1.9? If you can't answer immediately, you understand the problem."

Each institution must configure which attributes to release to which service providers. This creates an O(n√óm) configuration problem where n = number of SPs and m = number of attributes.

3.3 The Scoped Affiliation Nightmare

eduPersonScopedAffiliation is particularly problematic. It encodes both affiliation type AND institution scope in a single value:

Valid eduPersonScopedAffiliation Values

student@university.edu faculty@university.edu staff@university.edu employee@university.edu member@university.edu affiliate@university.edu alum@university.edu library-walk-in@university.edu

But what about "adjunct faculty"? "Visiting researcher"? "Emeritus professor"? The controlled vocabulary is inadequate for real-world complexity.

7. Security Considerations

Proponents of SAML often cite security as an advantage. The data tells a different story.

7.1 Attack Surface Comparison

SAML Attack Vectors

  • XML Signature Wrapping (20+ variants)
  • XML External Entity (XXE) Injection
  • XML Bomb / Billion Laughs DoS
  • Assertion Replay
  • Metadata Tampering
  • Certificate Substitution
  • Comment Injection
  • Schema Poisoning

OAuth 2.0 / OIDC Attack Vectors

  • Authorization Code Interception (mitigated by PKCE)
  • Token Theft (mitigated by short expiry)
  • Open Redirect (mitigated by exact redirect matching)
  • CSRF (mitigated by state parameter)

All major vectors have standardized mitigations

7.2 Cryptographic Agility

SAML's tight coupling of cryptographic operations with XML processing makes algorithm upgrades difficult. Many deployments still use SHA-1, deprecated since 2011:

Cryptographic Algorithm Usage in Production (2024 Survey)

SHA-1 23% of SAML IdPs
RSA-1024 8% of SAML IdPs
SHA-256+ 100% of modern OAuth providers

9. Conclusion

The data is unambiguous: SAML-based academic identity federation has become a liability. The complexity tax imposed by Shibboleth, OpenAthens, and eduPerson schemas actively harms educational technology adoption.

While these systems solved real problems in 2005, the industry has evolved. Modern OAuth 2.0/OIDC implementations offer:

  • 10x faster integration times
  • 2.3x lower total cost of ownership
  • 94% vs 52% first-attempt success rates
  • Dramatically smaller attack surface
  • Universal developer tooling and documentation

It is time for higher education to acknowledge what industry learned years ago: the age of SAML has passed. The path forward is clear.
